Data table

A data table is a display type optimized for large volumes of data, allowing users to analyze the data as a whole.

Usage
Status & changelog




Common alternative names

Data array, data grid, information table




Questions, comments, feedback?

Leave a comment in Figma or reach out to us on Slack.

#base-design-support







Anatomy

A data table usually contains a header row, zebra-striped rows, columns, and cells.

Header row

The header row has the same font size featured in the rest of the table but at a bolder weight. It also remains fixed. When a user scrolls down, the header row will continue to be visible, which provides the user with valuable contextual info about the data they’re looking for.




Rows

This table distinguishes between rows using zebra stripes. The first row after the header has a gray background, and the following row has a white background, and so on.

On hover, the background for the entire row of cells gets one shade darker. 

The header remains fixed when scrolling. If the table has a header column and a header row, then the header column remains fixed for horizontal scroll only.




Columns

If a user hovers on a column header, the column goes into a hover state, where each cell’s background becomes a shade darker. If applicable, a sort icon is also visible on the column header that is being hovered over.

Cell
When to Use

Use a data table when you need to analyze many rows of data. Features like infinite scrolling, filtering, sorting, and searching make analyzing large volumes of data easier.

Use for:

Large data sets requiring intensive analysis, often involving sorting, filtering, and batch actions.

Don't use for:

Small data sets, typically displayed as list items, which significantly benefit from extensive custom styling.

Note: This component has stricter requirements than other Base Web components. It doesn't support style overrides and requires data to adhere to strict column schemas. Unlike other tables, it doesn't support cell/column spans or dynamic row heights. Consider this more similar to Excel than a standard table.

Data types
Behavior
Filters

For applicable data types, when a user hovers on a column header, the column goes into a hover state. Each cell’s background becomes a shade darker, and a sort icon becomes visible on the right-hand side of the column header. The only data type that doesn't currently support sorting is “string.”

In the example shown above, there’s plenty of space for the icon to the right of the header. However, there may be cases in which the column header text takes up the full width of the header cell. In those cases, the text will appear truncated, and the chevron hover will cover up some of the text. Let’s use the Boolean column as an example:

First click: sorts descending, the icon changes from gray to black and remains even after the user hovers away to indicate a sort rule applied to that column.

Second click: sorts ascending, chevron icon points up, is black, and remains even after the user hovers away to indicate that there’s a sort rule applied to that column.

Third click: sort is removed, chevron points down, goes back to gray, and disappears when the user hovers away.

Search

Search is a great way to filter a table by any arbitrary string of characters. All non-numerical columns will be searchable. The search bar (small, enabled, label, and hint hidden) can be placed to the top left of the table, 12 padding.

The search bar itself will function as per Base UI specs. When a search is triggered, the searched-for string is highlighted in the result using bold font. See below the “category” column:

Batch actions

If your user can select multiple rows in a table and choose between multiple actions, then batch actions may be a feature worth offering. Examples of batch actions: download, upload, save, and share. A table with batch actions enabled will have checkboxes for all rows in the first columns.




Mousing over a single row in a table with batch actions enabled triggers batch action buttons to the far right of the row on hover. This may cover up some of the content of the table. Feel free to disable this feature if it doesn’t work with your table.

Clicking on the checkbox on the header cell allows user to quickly select or deselect all visible rows. Checking a single row triggers the batch actions header -- a collection of either icon or text buttons that replaces the search and filtering UI. If the user is considering a batch action, then the search and filtering UI become redundant.

Filters

A small pill-type button, placed on EITHER the top left of the table OR to the immediate right side of the search bar, can trigger advanced filtering.

You don’t have to use this particular trigger mechanism to use the filtering component. If a better design solution exists, feel free to deviate from this. This particular placement is merely a suggestion.

Clicking the “add filter” button triggers a dropdown showing you a list of columns one can filter by. A small icon to the left of each list item indicates the data type of that column.

Selecting a column triggers the appropriate filtering component. Here’s an example of a “date/time” filter component. Not all parts of this design are currently in production--the histograms are planned to be released at a later date.

Once the user defines filter criteria and clicks “apply,” a filter summary tag item appears to the right side of the add filter button.

Users can add as many filters as they like. Filter summary tags will stack. Users can delete filters by clicking on the “x.” The future version will support click-tag-to-edit-filter functionality as well.
