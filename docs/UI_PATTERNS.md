## A. Navigation & Hierarchy

### 핵심 원칙

정보 위계가 깊어질수록 사용자는 **"내가 지금 어디에 있는가"**를 잃기 쉽다. 네비게이션 패턴의 목적은 위치 감각을 유지하면서 원하는 곳에 빠르게 도달하게 하는 것이다.

### 네비게이션 컴포넌트 선택 판단 트리

```
이 화면의 정보 구조에 어떤 네비게이션을 사용해야 하는가?

1. 최상위 섹션 이동인가? (앱의 주요 영역 간 전환)
   → YES: Global Navigation (탑바 또는 사이드바)
   → NO: 계속

2. 같은 페이지 안에서 하위 뷰를 전환하는가?
   → YES: Tabs (가로 또는 세로)
   → NO: 계속

3. 현재 위치의 경로를 보여줘야 하는가? (깊은 계층 탐색)
   → YES: Breadcrumb
   → NO: 계속

4. 긴 페이지에서 특정 섹션으로 점프해야 하는가?
   → YES: Anchor Navigation (페이지 내 목차)
   → NO: 별도 네비게이션 불필요 (콘텐츠 흐름으로 충분)
```

### Global Navigation: 탑바 vs 사이드바

| 기준 | 탑바 (Top Navigation) | 사이드바 (Side Navigation) |
|------|:--------------------:|:------------------------:|
| 최상위 메뉴 개수 | 5개 이하 | 6개 이상 |
| 메뉴 아이템에 아이콘 필요 | 아이콘 선택적 | 아이콘 필수 (축소 시 아이콘만 표시) |
| 2-depth 이상 하위 메뉴 | 드롭다운으로 처리 (복잡하면 한계) | 자연스러운 트리 구조 가능 |
| 가로 공간이 중요한 콘텐츠 | O (전체 너비 사용 가능) | X (사이드바가 가로 공간 차지) |
| 콘텐츠가 도구/대시보드 성격 | 탑바로 부족할 수 있음 | O (SaaS, 어드민 패널에 적합) |
| 모바일 전환 | 햄버거 메뉴 | 하단 탭바 또는 햄버거 메뉴 |

```
판단 트리:

1. 주 메뉴 항목이 6개 이상인가?
   → YES: 사이드바
   → NO: 계속

2. 2-depth 이상의 하위 메뉴가 있는가?
   → YES: 사이드바 (트리 구조)
   → NO: 계속

3. 콘텐츠가 가로 전체 너비를 필요로 하는가? (에디터, 대시보드 차트 등)
   → YES: 탑바 (또는 축소 가능한 사이드바)
   → NO: 어느 쪽이든 가능. 제품 성격에 따라 선택.
```

### Tabs: 가로 vs 세로

| 기준 | 가로 탭 (Horizontal) | 세로 탭 (Vertical) |
|------|:-------------------:|:-----------------:|
| 탭 개수 | 2~6개 | 6개 이상 가능 |
| 탭 레이블 길이 | 짧은 텍스트 (1~3단어) | 긴 텍스트 가능 |
| 콘텐츠 영역 형태 | 아래로 넓게 | 우측으로 넓게 |
| 모바일 대응 | 스크롤 가능 탭바 | 가로 탭으로 전환 또는 아코디언 |
| 익숙함 | 매우 높음 (웹 표준) | 설정 페이지에서 주로 사용 |

### 다중 위계 조합 규칙

페이지에 2개 이상의 네비게이션 위계가 필요할 때, **상위는 가로, 하위는 세로** 원칙을 따른다.

```
위계 1 (가장 상위):  Global Nav (탑바 또는 사이드바)
    │
위계 2:             가로 탭 (Horizontal Tabs)
    │
위계 3 (있다면):     세로 탭 (Vertical Tabs) 또는 앵커 네비게이션
```

**규칙:**

1. **같은 축의 네비게이션을 중복하지 않는다**
   - 가로 탭 아래에 또 가로 탭 → 금지 (어디가 상위인지 구분 불가)
   - 사이드바 안에 또 사이드바 → 금지

2. **최대 3단계까지만 허용**
   - 3단계를 넘어가면 정보 구조 자체를 재설계해야 한다
   - 4단계가 필요하다고 느끼면: 페이지를 분리하거나, 드릴다운 패턴 사용

3. **위계 간 시각적 구분을 명확히**
   - 상위 네비게이션: 고정 (sticky), 배경색 있음
   - 하위 네비게이션: 콘텐츠와 같은 레벨, 배경 없거나 미묘한 구분

| 조합 | 적합한 상황 | 예시 |
|------|-----------|------|
| 탑바 + 가로 탭 | 단순 구조의 SaaS | 대시보드 탑바 → 각 메뉴의 하위 뷰를 탭으로 |
| 사이드바 + 가로 탭 | 복잡한 어드민/툴 | 사이드바로 주요 섹션 → 탭으로 세부 뷰 전환 |
| 사이드바 + 세로 탭 | 설정/관리 페이지 | 사이드바로 설정 카테고리 → 세로 탭으로 세부 항목 |
| 탑바 + 가로 탭 + 앵커 | 문서/레퍼런스 | 탑바 → 카테고리 탭 → 긴 페이지 내 목차 |

### Breadcrumb 사용 규칙

| 규칙 | 설명 |
|------|------|
| 사용 조건 | 정보 계층이 **3단계 이상**이고 사용자가 상위로 돌아갈 필요가 있을 때 |
| 미사용 | 계층이 2단계 이하이면 불필요 (탭이나 뒤로가기로 충분) |
| 위치 | 페이지 콘텐츠 최상단, 제목 위 |
| 현재 페이지 | 마지막 항목으로 표시하되 링크 아님 (텍스트만) |
| 생략 | 5단계 이상이면 중간 단계를 `...`으로 생략 가능 |
| 탭과 병용 | 가능. Breadcrumb은 "어디서 왔는가", 탭은 "지금 무엇을 보는가" |

## B. Overlay 패턴

### 핵심 원칙

오버레이는 **현재 맥락을 유지하면서** 추가 정보/작업을 제공한다. 하지만 남용하면 사용자가 "원래 뭘 하고 있었지?"를 잃는다. **가능하면 오버레이 없이 해결하고, 필요할 때만 최소한으로 사용한다.**

### Overlay 유형 선택 판단 트리

```
현재 화면에서 벗어나지 않고 추가 정보/작업을 보여줘야 한다.
어떤 오버레이를 사용하는가?

1. 사용자의 결정이 필요한가? (확인/취소, 삭제 확인 등)
   → YES: Dialog (Alert Dialog)
   → NO: 계속

2. 입력/편집 작업인가?
   → YES: 계속 3번
   → NO: 계속 5번

3. 입력 필드가 3개 이하인가?
   → YES: Dialog (간단한 폼)
   → NO: 계속 4번

4. 입력이 복잡하거나 콘텐츠가 길어질 수 있는가?
   → YES: Drawer (사이드) 또는 별도 페이지
   → NO: Dialog (중간 복잡도 폼)

5. 보조 정보를 보면서 메인 콘텐츠도 참조해야 하는가?
   → YES: Drawer (사이드) — 메인 콘텐츠가 일부 보임
   → NO: 계속 6번

6. 콘텐츠가 짧은 안내/확인인가?
   → YES: Dialog
   → NO: Drawer 또는 별도 페이지 검토
```

### Modal vs Drawer vs 페이지 이동

| 기준 | Dialog (Modal) | Drawer (사이드/하단) | 페이지 이동 |
|------|:--------------:|:-------------------:|:----------:|
| 콘텐츠 양 | 적음 (1~2 스크롤 이내) | 중간 (스크롤 가능) | 많음 |
| 메인 콘텐츠 참조 | 필요 없음 (포커스 완전 전환) | **필요함** (옆에서 참조) | 필요 없음 |
| 작업 복잡도 | 단순 (확인, 짧은 입력) | 중간 (폼, 리스트 탐색) | 복잡 (멀티스텝, 복합 입력) |
| 되돌리기 | 쉬움 (ESC, 바깥 클릭) | 쉬움 (ESC, 바깥 클릭) | 뒤로가기 |
| URL 필요 | 보통 불필요 | 선택적 | **필수** (공유/북마크 가능) |
| 중첩 | 금지 (아래 참조) | 가능하지만 최소화 | 자연스러움 |

### Drawer 방향 선택

| 방향 | 적합한 상황 | 일반적 너비/높이 |
|------|-----------|:-------------:|
| **우측** (기본) | 상세 정보 패널, 편집 폼, 설정 | 320~480px (데스크탑) |
| **좌측** | 네비게이션, 필터 패널 | 280~360px |
| **하단** | 모바일 액션 메뉴, 짧은 선택지 | 화면의 40~70% 높이 |

```
Drawer 방향 선택:

1. 콘텐츠가 네비게이션/필터 성격인가?
   → YES: 좌측 (네비게이션 관행)

2. 콘텐츠가 상세 정보/편집 성격인가?
   → YES: 우측 (마스터-디테일 패턴의 디테일 위치)

3. 모바일에서 짧은 선택지를 보여주는가?
   → YES: 하단 (Bottom Sheet)
```

### 오버레이 중첩 규칙

| 규칙 | 설명 |
|------|------|
| **Modal 위에 Modal 금지** | 정보 구조를 재설계하거나 첫 번째 모달을 닫고 새로운 것을 열어야 한다 |
| **Modal 위에 Dialog 가능** | 확인 다이얼로그만 허용 (예: "저장하지 않고 닫으시겠습니까?") |
| **Drawer 위에 Modal 가능** | Drawer 작업 중 확인이 필요할 때 (예: 삭제 확인) |
| **Drawer 위에 Drawer 금지** | 2단 Drawer는 혼란. 페이지 이동으로 전환 |
| **최대 중첩: 2단계** | 어떤 조합이든 3단 중첩은 금지. 정보 구조 재설계 필요 |

```
중첩 허용 판단:

1. 이미 오버레이가 열려 있는가?
   → NO: 자유롭게 열기
   → YES: 계속

2. 새로 열려는 것이 확인 다이얼로그(Alert Dialog)인가?
   → YES: 허용 (위에 열기)
   → NO: 금지. 대안 검토:
     - 현재 오버레이를 닫고 새 것을 열기
     - 현재 오버레이 안에서 단계 전환 (스텝 패턴)
     - 페이지 이동으로 전환
```

### 오버레이 닫기 패턴

| 닫기 수단 | Dialog | Drawer | 비고 |
|----------|:------:|:------:|------|
| ESC 키 | O | O | 항상 지원 |
| 바깥 영역 클릭 (Backdrop) | O (기본) | O (기본) | 파괴적 작업 중에는 비활성화 고려 |
| 명시적 닫기 버튼 (X) | O | O | 항상 표시 |
| 스와이프 (모바일) | — | O (하단 Drawer) | 바텀시트에서 아래로 스와이프 |
| 완료/저장 액션 | O | O | 작업 완료 시 자동 닫기 |

**파괴적 작업 중 닫기 방지**: 사용자가 폼을 수정 중이고 저장하지 않았을 때, 바깥 클릭이나 ESC로 닫으면 데이터 손실 위험. 이 경우 확인 다이얼로그를 먼저 표시한다 ("저장하지 않은 변경사항이 있습니다. 닫으시겠습니까?").

## C. Data Display

### 핵심 원칙

데이터 표시 패턴의 선택은 **"사용자가 이 데이터로 무엇을 하는가"**에 달려있다. 비교·정렬이 중요하면 테이블, 탐색·발견이 중요하면 카드, 시간 흐름이 중요하면 타임라인.

### 데이터 표시 패턴 선택 판단 트리

```
이 데이터를 어떤 형태로 보여줘야 하는가?

1. 항목 간 비교가 중요한가? (같은 속성을 나란히 비교)
   → YES: Table
   → NO: 계속

2. 항목별로 시각적 프리뷰가 중요한가? (이미지, 썸네일, 요약)
   → YES: Card Grid / Card List
   → NO: 계속

3. 시간 순서가 핵심 맥락인가?
   → YES: Timeline
   → NO: 계속

4. 항목이 단순한 텍스트 목록인가?
   → YES: List (Simple List)
   → NO: Table 또는 Card 중 데이터 특성에 맞게 선택
```

### Table vs Card

| 기준 | Table | Card |
|------|:-----:|:----:|
| 주 목적 | 비교, 정렬, 필터링 | 탐색, 발견, 개별 항목 상세 |
| 데이터 속성 수 | 많음 (5개 이상 열) | 적음 (3~5개 핵심 정보) |
| 시각적 요소 | 텍스트 위주 | 이미지/아이콘/그래프 포함 |
| 밀도 | 높음 (한 화면에 많은 항목) | 낮음 (여백과 시각적 계층) |
| 대량 액션 | 체크박스 + 벌크 액션에 적합 | 개별 액션에 적합 |
| 모바일 | 적응 어려움 (반응형 전략 필요) | 자연스러운 스택 레이아웃 |

### Table 설계 규칙

| 규칙 | 설명 |
|------|------|
| **정렬 가능 열 표시** | 정렬 가능한 열은 헤더에 정렬 아이콘 표시. 현재 정렬 방향 명시 |
| **고정 열** | 식별자 열(이름, ID)은 좌측 고정 (수평 스크롤 시에도 보임) |
| **열 우선순위** | 좌측에 가장 중요한 열. 좌→우로 중요도 감소 |
| **행 높이 통일** | 같은 테이블 안에서 행 높이 혼합 금지 |
| **빈 셀** | `—`(em dash) 표시. 공백으로 두지 않는다 |
| **숫자 정렬** | 숫자는 우측 정렬 (자릿수 비교 용이) |
| **텍스트 정렬** | 텍스트는 좌측 정렬 |
| **액션 열** | 우측 끝에 배치. 고정 너비 |
| **페이지네이션** | 50개 이상 항목 시 페이지네이션 또는 무한 스크롤 |

### Table 반응형 전략

```
화면이 좁아져 테이블이 넘칠 때:

1. 열이 6개 이하인가?
   → 열 너비 축소 + 텍스트 말줄임으로 대응 가능할 수 있음

2. 열이 7개 이상이거나 축소로 해결 안 되는가?
   → 옵션 A: 수평 스크롤 (식별자 열 고정)
   → 옵션 B: 카드 리스트로 전환 (Adapt 패턴)
   → 옵션 C: 우선순위 낮은 열 숨김 (Prioritize 패턴)

3. 모바일 (< md)에서는?
   → 카드 리스트 전환 권장. 수평 스크롤 테이블은 최후 수단.
```

> 반응형 패턴(Reflow, Adapt, Prioritize 등)은 [INTERACTION_DESIGN.md §G](./INTERACTION_DESIGN.md#g-responsive-strategy) 참고.

### Card Grid 설계 규칙

| 규칙 | 설명 |
|------|------|
| **일관된 카드 높이** | 같은 그리드 내 카드 높이 통일 권장 (이미지 비율 고정, 텍스트 말줄임) |
| **카드 내 정보 계층** | 이미지/썸네일 → 제목 → 부가 정보 → 액션 (위→아래 순) |
| **카드 액션** | 카드 전체 클릭 가능 (상세 이동) + 개별 액션 버튼은 우하단 또는 메뉴 |
| **그리드 열 수** | 데스크탑: 3~4열, 태블릿: 2~3열, 모바일: 1~2열 |
| **간격** | 카드 간: `gap-4` (16px) 또는 `gap-6` (24px). 그룹 간 간격보다 크지 않게 |

### 필터/정렬 UI 패턴

| 패턴 | 적합한 상황 | 위치 |
|------|-----------|------|
| **인라인 필터** | 필터 조건 1~3개, 항상 보여야 함 | 테이블/리스트 상단, 수평 배치 |
| **필터 바** | 필터 조건 4~6개, 자주 변경 | 콘텐츠 영역 상단, 줄바꿈 허용 |
| **필터 패널 (사이드)** | 필터 조건 7개 이상, 복합 조건 | 좌측 사이드 패널 (데스크탑), 바텀시트(모바일) |
| **정렬 드롭다운** | 정렬 기준 변경 | 테이블 헤더 또는 필터 바 우측 |

```
필터 UI 선택:

1. 필터 조건이 3개 이하인가?
   → YES: 인라인 필터

2. 필터 조건이 4~6개인가?
   → YES: 필터 바

3. 필터 조건이 7개 이상이거나 범위/다중선택 등 복합 조건이 있는가?
   → YES: 필터 패널 (사이드)

모바일에서는:
- 인라인 필터: 수평 스크롤 칩(chip) 형태
- 필터 바 이상: "필터" 버튼 → 바텀시트로 전체 필터 표시
```

### 검색 패턴

| 패턴 | 적합한 상황 |
|------|-----------|
| **인라인 검색** | 현재 목록/테이블 내에서 필터링. 별도 결과 페이지 없음 |
| **글로벌 검색** | 앱 전체에서 검색. 전용 결과 페이지 또는 드롭다운 |
| **검색 + 필터 병행** | 텍스트 검색으로 좁힌 후 필터로 추가 조건 |

## D. Form & Input

### 핵심 원칙

폼의 목적은 **사용자가 필요한 데이터를 최소한의 노력으로 정확하게 입력**하게 하는 것이다. 모든 폼 설계 결정은 "이것이 사용자의 입력 부담을 줄이는가?"로 판단한다.

### 폼 구조 선택 판단 트리

```
이 입력 작업에 어떤 폼 구조를 사용해야 하는가?

1. 입력 필드가 6개 이하인가?
   → YES: 단일 폼 (Single Form)
   → NO: 계속

2. 입력 항목이 명확한 단계로 나뉘는가? (예: 개인정보 → 결제 → 확인)
   → YES: 멀티스텝 위자드 (Wizard)
   → NO: 계속

3. 입력 항목을 논리적 그룹으로 분류할 수 있는가?
   → YES: 섹션 분리 폼 (Grouped Form)
   → NO: 멀티스텝 위자드 권장 (7개 이상 필드가 그룹화도 안 되면 UX 재설계 필요)
```

### 단일 폼 (Single Form)

| 규칙 | 설명 |
|------|------|
| 필드 개수 | 6개 이하 |
| 레이아웃 | 1열 세로 스택이 기본. 관련 필드(이름+성, 시작일+종료일)만 가로 병렬 |
| 레이블 위치 | 필드 위 (top-aligned). 좌측 정렬(left-aligned)은 가로 공간이 넉넉할 때만 |
| 액션 버튼 | 폼 하단. Primary(제출) 좌측, Secondary(취소) 우측. 또는 Primary만 |
| 필수 표시 | 대부분 필수이면 선택 필드에 "(선택)" 표시. 대부분 선택이면 필수 필드에 `*` 표시 |

### 섹션 분리 폼 (Grouped Form)

| 규칙 | 설명 |
|------|------|
| 필드 개수 | 7~15개 |
| 그룹 구분 | 섹션 제목 (`typography-18-semibold`) + Divider |
| 그룹 간 간격 | 섹션 간 `gap-8` (32px) 이상 |
| 그룹 내 간격 | 필드 간 `gap-4` (16px) ~ `gap-6` (24px) |
| 저장 | 전체 폼에 하나의 저장 버튼 (섹션별 저장 아님) |

### 멀티스텝 위자드 (Wizard)

| 규칙 | 설명 |
|------|------|
| 사용 조건 | 필드 15개 이상, 또는 단계별 검증/의존성이 있을 때 |
| 단계 수 | 3~5단계 권장. 7단계 이상이면 정보 구조 재설계 |
| 진행 표시 | 스텝 인디케이터 (현재 단계 + 전체 단계 수) 항상 표시 |
| 뒤로가기 | 이전 단계로 언제든 돌아갈 수 있어야 함 (입력값 유지) |
| 단계별 저장 | 각 단계 이동 시 입력값 보존 (브라우저 뒤로가기에도) |
| 최종 확인 | 마지막 단계에서 입력 내용 요약 → 확인 → 제출 |

### 인라인 편집 vs 편집 모드

```
기존 데이터를 수정해야 한다. 어떤 방식을 사용하는가?

1. 수정할 필드가 1~2개인가?
   → YES: 인라인 편집 (해당 필드만 편집 모드 전환)
   → NO: 계속

2. 수정할 필드가 여러 개이고, 동시에 수정하는 경우가 많은가?
   → YES: 편집 모드 전환 (보기→편집 모드 토글)
   → NO: 인라인 편집 (개별 필드 클릭으로 편집)

3. 수정이 즉시 반영되어야 하는가? (자동 저장)
   → YES: 인라인 편집 + 자동 저장 (편집 완료 시 blur에서 저장)
   → NO: 명시적 저장 버튼
```

| 기준 | 인라인 편집 | 편집 모드 전환 |
|------|:----------:|:-------------:|
| 수정 빈도 | 가끔, 개별 필드 | 자주, 여러 필드 동시 |
| 저장 방식 | 자동 저장 (blur 시) 또는 체크 아이콘 | 명시적 저장/취소 버튼 |
| 전환 트리거 | 필드 클릭/더블클릭 또는 연필 아이콘 | "편집" 버튼 |
| 취소 | ESC 또는 바깥 클릭 | "취소" 버튼 |
| 시각적 변화 | 해당 필드만 입력 스타일로 전환 | 전체 영역이 폼 스타일로 전환 |

### 폼 레이아웃 규칙

| 규칙 | 설명 |
|------|------|
| **1열이 기본** | 좌→우 시선 이동을 최소화. 세로로 읽어 내려가는 것이 가장 빠름 |
| **2열은 관련 필드만** | 이름+성, 시작일+종료일 등 의미적으로 한 쌍인 것만 가로 배치 |
| **3열 이상 금지** | 폼에서 3열은 시선 분산. 그리드 레이아웃이 필요하면 테이블이 더 적합 |
| **레이블은 필드 위** | top-aligned 레이블이 가장 빠른 완성 시간 (연구 기반) |
| **도움말 텍스트** | 필드 아래, 에러 메시지보다 위. `typography-13-regular`, `sys-text-on-bright-500` |
| **필드 너비는 예상 입력 길이에 비례** | 우편번호 필드가 주소 필드와 같은 너비이면 불안감 |

### 입력 컴포넌트 선택

```
이 데이터를 어떤 입력 컴포넌트로 받아야 하는가?

1. 자유 텍스트인가?
   → 1줄: Text Input
   → 여러 줄: Textarea

2. 미리 정해진 선택지 중 고르는가?
   → 2~5개: Radio Group (단일 선택) 또는 Checkbox Group (복수 선택)
   → 6개 이상: Select (Dropdown) 또는 Combobox (검색 가능 드롭다운)

3. 켜기/끄기 (on/off) 설정인가?
   → 즉시 적용: Switch (Toggle)
   → 제출 시 적용: Checkbox

4. 날짜/시간인가?
   → Date Picker (달력 UI)

5. 범위 값인가? (볼륨, 가격 범위)
   → Slider (Range)

6. 파일인가?
   → File Upload (드래그앤드롭 영역 + 버튼)
```

| 선택지 수 | 단일 선택 | 복수 선택 |
|:---------:|----------|----------|
| 2~3개 | Radio Group (항상 표시) | Checkbox Group |
| 4~5개 | Radio Group 또는 Select | Checkbox Group |
| 6~10개 | Select (Dropdown) | Multi-select Dropdown |
| 11개 이상 | Combobox (검색 가능) | Multi-select + 검색 |

## E. Page Composition

### 핵심 원칙

페이지는 컴포넌트의 단순 나열이 아니라, **사용자의 목적에 맞춘 정보 흐름**이다. 페이지 구성은 "사용자가 이 페이지에서 무엇을 달성하려 하는가?"에서 시작한다.

### 페이지 유형과 레이아웃 패턴

| 유형 | 사용자 목적 | 레이아웃 패턴 | 예시 |
|------|-----------|:----------:|------|
| **대시보드** | 현황 파악 + 빠른 진입점 | 그리드 (위젯/카드 배치) | 홈, 통계 |
| **목록** | 탐색 + 필터링 + 선택 | 상단 필터 + 리스트/테이블/카드 | 사용자 목록, 상품 목록 |
| **상세** | 하나의 항목을 깊이 파악 | 헤더 영역 + 탭 또는 섹션 | 사용자 프로필, 상품 상세 |
| **편집** | 데이터 입력/수정 | 폼 레이아웃 (1~2열) | 설정, 프로필 편집 |
| **마스터-디테일** | 목록 탐색 + 선택 항목 상세 | 좌측 리스트 + 우측 디테일 | 이메일, 채팅 |
| **빈 상태** | 시작 유도 | 중앙 정렬 CTA | 첫 사용, 검색 결과 없음 |

### 마스터-디테일 패턴

좌측에서 항목을 선택하면 우측에 상세가 나타나는 패턴.

| 규칙 | 설명 |
|------|------|
| 마스터 영역 너비 | 280~400px 고정. 콘텐츠에 따라 조정하되 1/3 이하 |
| 디테일 영역 | 나머지 너비 사용 (`flex-1`) |
| 선택 상태 표시 | 마스터 목록에서 현재 선택 항목 하이라이트 |
| 미선택 시 | 디테일 영역에 빈 상태 ("항목을 선택하세요") 또는 첫 항목 자동 선택 |
| 모바일 | 2패널 → 1패널 전환. 마스터 전체 화면 → 선택 시 디테일 전체 화면 (뒤로가기로 복귀) |
| 구분선 | 마스터와 디테일 사이에 수직 Divider 또는 그림자 |

### 대시보드 패턴

| 규칙 | 설명 |
|------|------|
| 위젯 그리드 | 12열 그리드 기반. 위젯은 3, 4, 6, 12열 단위로 배치 |
| 정보 우선순위 | 상단/좌측에 가장 중요한 위젯. 좌상단이 시선 시작점 |
| KPI 카드 | 최상단 수평 배치 (핵심 지표 3~5개). 아이콘 + 숫자 + 변화율 |
| 차트 배치 | KPI 아래. 넓은 차트(트렌드)를 위, 좁은 차트(비율)를 아래/옆 |
| 최근 활동 | 우측 또는 하단. 시간순 리스트 |
| 새로고침 | 자동 갱신 주기 표시 + 수동 새로고침 버튼 |

### 페이지 상태 전이

모든 데이터 기반 페이지는 아래 4가지 상태를 거친다. 각 상태에 대한 UI가 반드시 설계되어야 한다.

```
                  데이터 요청
  ┌─────────┐  ──────────▶  ┌──────────┐
  │         │               │          │
  │  Empty  │               │ Loading  │
  │         │               │          │
  └─────────┘               └────┬─────┘
       ▲                         │
       │                    성공  │  실패
       │                    ┌────┘  └────┐
       │                    ▼            ▼
       │              ┌──────────┐  ┌─────────┐
       │              │          │  │         │
       └─────────────│ Populated │  │  Error  │
        데이터 삭제    │          │  │         │
                      └──────────┘  └─────────┘
```

| 상태 | UI 처리 | 필수 여부 |
|------|--------|:--------:|
| **Empty** | 빈 상태 UI (아이콘 + 메시지 + CTA) | **필수** |
| **Loading** | 스켈레톤 또는 스피너 | **필수** |
| **Populated** | 정상 콘텐츠 | **필수** |
| **Error** | 에러 메시지 + 재시도 버튼 | **필수** |

> 각 상태의 상세 설계는 [INTERACTION_DESIGN.md §D (Loading & Empty States)](./INTERACTION_DESIGN.md#d-loading--empty-states)와 [INTERACTION_DESIGN.md §C (Feedback 패턴)](./INTERACTION_DESIGN.md#c-feedback-패턴) 참고.

**원칙: 4개 상태 중 하나라도 설계하지 않으면 불완전한 페이지다.** 데이터가 있을 때만 설계하고 로딩/빈/에러 상태를 빠뜨리는 것은 가장 흔한 실수.

### 상세 페이지 구성 패턴

| 영역 | 내용 | 위치 |
|------|------|------|
| **헤더** | 제목, 상태 배지, 주요 액션 버튼 | 최상단. Sticky 선택적 |
| **메타 정보** | 생성일, 작성자, 카테고리 등 | 헤더 아래 또는 헤더 우측 |
| **탭 (선택)** | 하위 뷰 전환 (개요/활동/설정 등) | 헤더 아래. Sticky 권장 |
| **본문** | 주 콘텐츠 (탭 콘텐츠 또는 단일 뷰) | 탭 아래 |
| **사이드바 (선택)** | 관련 정보, 속성 패널 | 우측. 데스크탑에서만 표시 |

```
상세 페이지에 탭이 필요한가?

1. 표시할 정보가 명확히 다른 카테고리로 나뉘는가?
   → YES: 탭 사용
   → NO: 계속

2. 모든 정보를 한 페이지에서 스크롤로 볼 수 있는가?
   → YES: 탭 불필요. 섹션 분리로 충분
   → NO: 탭 사용하여 각 카테고리 분리

3. 각 탭의 데이터가 독립적으로 로딩되는가?
   → YES: 탭 전환 시 해당 탭만 로딩 (성능 최적화)
   → NO: 전체 로딩 후 탭은 클라이언트에서 전환
```

### 페이지 간 전환 패턴

| 패턴 | 적합한 상황 | 예시 |
|------|-----------|------|
| **하드 네비게이션** | 완전히 다른 맥락으로 이동 | 목록 → 상세 (URL 변경) |
| **탭 전환** | 같은 맥락 내 뷰 변경 | 상세 페이지의 "개요" ↔ "활동" |
| **오버레이** | 현재 맥락 유지 + 부가 작업 | 목록에서 "빠른 편집" 모달 |
| **드릴다운** | 계층 깊이 탐색 | 카테고리 → 서브카테고리 → 항목 |
| **뒤로가기** | 이전 위치 복귀 | 상세 → 목록 (스크롤 위치 복원) |

**스크롤 위치 복원 규칙**: 사용자가 목록에서 항목을 클릭해 상세를 본 후 뒤로가기하면, **원래 스크롤 위치로 복원**해야 한다. 목록 최상단으로 돌아가면 사용자가 "내가 어디까지 봤지?"를 다시 찾아야 한다.

## 패턴 선택 가이드 (요약)

새 화면을 설계할 때 아래 순서로 판단한다.

```
1. 이 화면의 사용자 목적은 무엇인가?
   → 현황 파악: 대시보드 패턴 → E절
   → 탐색/선택: 목록 패턴 → E절 + C절
   → 깊은 파악: 상세 패턴 → E절
   → 입력/수정: 편집 패턴 → E절 + D절

2. 네비게이션 구조는?
   → 위계 수와 메뉴 개수 → A절 판단 트리
   → 다중 위계 필요하면 조합 규칙 확인

3. 현재 페이지 안에서 추가 작업이 필요한가?
   → YES: 오버레이 판단 트리 → B절
   → NO: 페이지 이동

4. 데이터를 어떤 형태로 보여줄 것인가?
   → 데이터 표시 패턴 선택 → C절

5. 사용자 입력이 필요한가?
   → 폼 구조 + 입력 컴포넌트 선택 → D절

6. 4가지 상태 (Empty/Loading/Populated/Error) 모두 설계했는가?
   → 하나라도 빠지면 불완전. → E절 상태 전이
```
